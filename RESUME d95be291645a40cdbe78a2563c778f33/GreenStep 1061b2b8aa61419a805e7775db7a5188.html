기술적 의사결정.

SSE

  • 실시간 알람 기능 Web Socket vs SSE
    • Polling 방식을 사용했을 경우
      • 방법: DeferredResult 클래스로 비동기 처리해 주며 구현
      • 장점: 소프트웨어 구현이 간편함
      • 단점: 반응 시간이 느리고 리소스를 많이 차지한다
    • WebSocket 방식을 사용했을 경우
      • 방법: websocket 패키지 설치 및 @EnableWebSocket를 이용하여 구현
      • 장점: 한번의 연결로 서버와 지속적인 연결이 가능해서 polling, Long polling 방식에서 사용되는 불필요한 커넥션 오버헤드(HTTP, TCP 연결 트래픽 등)를 줄일 수 있음
      • 단점: 서버와 클라이언트 간의 Socket 연결을 유지하는 것 자체에 비용이 발생.
    • Server Sent Event 방식을 사용했을 경우
      • 방법: SseEmitter 클래스를 이용하여 구현
      • 장점: 웹 소켓의 역할을 어느 정도 대체 가능하면서 더 가볍다.
      • 단점: 양방향 통신이 아닌 서버에서 클라이언트로만 신호를 보내 줄 수 있다.
  • 서버에서 클라이언트로 신호를 보내주는 실시간 알림 기능을 구현하는 것이 목적이었기에 양방향 통신인 Web Socket 보단 단방향 통신인 SSE가 목적성으로나 트래픽 부하 등의 이유로 더 적합하다고 판단하여 SSE으로 결정하게 되었다.

트러블슈팅.

1. 이미지 리사이징 기능 구현 30-60% 이미지 크기 감축

  • 문제상황: 이미지 크기로 인한 S3버킷 부하를 줄이기 위해 이미지 리사이징 기능 결정

  • 시도한 방법:
    1. 받은 Base64를 InputStream으로 변환하여 ImageIO.read를 통해 받은 이미지의 길이와 넓이를 확인 후 가로세로 최솟값이 320이 되도록 리사이징 함.
    1. InputStream은 BufferedImage로 변환 시 null이 뜨는 오류 수정
      • 시도한 로직에서 InputStream이 ImageIO의 read 함수를 거치고 나면 InputStream이 null이 뜨며 BufferedImage로의 변환이 실패하는데 이는 Image.IO에 대한 이해도 부족으로 인한 오류였다. BufferedImage를 받아와 ImageIO.read 함수를 사용하지 않고 저장으로 넘겨 정상 저장 완료하였다.

  • 해결한 방법:
    • 결론: 버킷에 저장되는 이미지는 최대 280KB까지, 저장되며 이는 원본에 비하여 약 30-60%의 이미지 크기 감축을 하였다. 이후 추가적인 렌더링의 성능 개선과 버킷 리소스 부하를 줄이기 위해서 CDN을 사용하여 서버 부하를 줄였다.
    • 해결한 코드
      public String getImgUrlBase64(String base64) throws IOException {
        String[] strings = base64.split(",");
        String extension = switch (strings[0]) { 
            case "data:image/jpeg;base64" -> ".jpeg";
            case "data:image/png;base64" -> ".png";
            case "data:image/webp;base64" -> ".webp";
            default -> ".jpg";
        };
        byte[] data = DatatypeConverter.parseBase64Binary(strings[1]);

        InputStream imgStream = new ByteArrayInputStream(data);
        InputStream resizedImg = checkImage(imgStream);

        ObjectMetadata metadata = new ObjectMetadata();
        metadata.setContentLength(resizedImg.available());
        metadata.setContentType("image/" + extension.substring(1));
        metadata.setCacheControl("public, max-age=31536000");

        long now = new Date().getTime();
        String fileName = now + "_" + UUID.randomUUID() + extension;
        amazonS3Client.putObject(new PutObjectRequest(bucket, fileName, resizedImg, metadata)
                .withCannedAcl(CannedAccessControlList.PublicRead));
        return amazonS3Client.getUrl(bucket, fileName).toString();
    }

    public InputStream checkImage(InputStream imgStream) throws IOException {
        BufferedImage inputImage = ImageIO.read(imgStream);
        int imageHeight = inputImage.getHeight(null);
        int imageWidth = inputImage.getWidth(null);
        if (imageHeight > 320 && imageWidth > 320) return resize(inputImage, imageWidth, imageHeight);
        else {
            ByteArrayOutputStream outStream = new ByteArrayOutputStream();
            ImageIO.write(inputImage,"png",outStream);
            return new ByteArrayInputStream(outStream.toByteArray());
        }
    }
    /* 리사이즈 실행 메소드 */
    public InputStream resize(BufferedImage inputImage, int width, int height) throws IOException {

        ImageSizeDto sizeDto = defineSize(height, width);
        Image resizedImage = inputImage.getScaledInstance(sizeDto.getWidth(),sizeDto.getHeight(),Image.SCALE_SMOOTH);

        BufferedImage outputImage = new BufferedImage(sizeDto.getWidth(), sizeDto.getHeight(), inputImage.getType());

        Graphics graphics2D = outputImage.getGraphics();
        graphics2D.drawImage(resizedImage, 0, 0, null); // 그리기
        graphics2D.dispose(); // 자원해제

        ByteArrayOutputStream os = new ByteArrayOutputStream();
        ImageIO.write(outputImage, "png", os);
        return new ByteArrayInputStream(os.toByteArray());
    }

    public ImageSizeDto defineSize(int height, int width) {
        if (height > width) {
            double ratio = (double)width / 320;
            int resizedHeight = (int) (height / ratio);
            return (new ImageSizeDto(resizedHeight, 320));
        }
        int ratio = height / 320;
        int resizedWidth = width / ratio;
        return (new ImageSizeDto(320, resizedWidth));
    }

2. N+1 문제 처리, 스레드 요청 처리 시간 45초 단축

  • 문제상황 : @GetMapping("admin/verification") api와 @GetMapping("profiles/feed") api에서 참조로 인한 n+1 문제 발생. 이로 인한 api 처리 속도 지연 이슈 발생

  • 해결한 방법 : Fetch Join 방식을 이용하여 n+1 문제 해결
    • 사용한 코드
      @Query("select s from SubmitMission s join fetch s.member order by s.createdAt asc")
    List<SubmitMission> findAllByOrderByCreatedAtAscFetchJoin();

@Query("select f from Feed f join fetch f.member where f.member = :member and f.onHide = :onHide order by f.createdAt desc ")
    List<Feed> findAllByMemberFetchJoin(@Param("member") Member member, @Param("onHide") Boolean onHide);

  • @GetMapping("admin/verification") Fetch Join 적용 후

    초당 데이터의 처리 양과 수신받는 데이터의 양, 전송되는 데이터의 양이 기존보다 168% 증가하였다.

    1,000개의 스레드의 요청을 처리해 주기까지 소모되는 시간이 기존 70초에서 26초까지 44초 단축되었다.

  • @GetMapping("profiles/feed") Fetch Join 적용 후

    초당 데이터의 처리양과 수신받는 데이터의 양, 전송되는 데이터의 양이 기존보다 17% 증가하였다.

    1,000개의 스레드의 요청을 처리해 주기까지 소모되는 시간이 기존 10초에서 9초까지 1초 단축되었다.

3. Histroy Table 생성

  • 문제상황 :

    관리자의 입장에서 유저의 정보들이 누락되는 경우 민원 처리나 클레임을 대처할 때, 어려움이 있을 수 있다. 그러한 점을 고려하여 미션 인증 내역을 따로 저장하는 기능을 구현했다. 그 과정에서 고려한 기능은 일반 테이블에 JPA로 자동 기록하게 하는 형식과 envers 라이브러리를 통한 내역 테이블 자동생성 두 기능을 비교하였다.

  • 시도한 방법 :
    • 일반 테이블에 JPA로 자동 기록하게 하는 형식
      • 방법: JPA @PrePersist 와 @PreUpdate를 통하여 테이블 자동 생성
      • 장점: 원하는 정보만을 선별하여 꼭 필요할 때에만 내역을 저장할 수 있다.
      • 단점: 자바class가 여러 개가 생성되며 단순 내역 기록에 사용되기에는 코드의 양이 많은 편이다.
    • envers 라이브러리
      • 방법: envers를 사용하여 @Audited와 @NotAudited로 테이블 내 정보를 선별하여 Rev파일 자동 생성
      • 장점: 이미 생성되어 있는 테이블에 어노테이션 @Audited로 테이블에 변경사항이 생길 때 자동으로 내역 테이블을 별도의 클래스 없이 생성한다.
      • 단점: 테이블 내에 있는 정보가 아니면 기록되지 않는다.
  • 결론 : 필요한 정보가 submitMission 테이블에 존재하기 때문에 envers 라이브러리를 사용하기로 결정.
